# Technical — Testing Strategy

Update as the system evolves.

- **Five test buckets.**
  - **Backend unit** — Nest's `Test.createTestingModule` with mocked providers, including the fake Anthropic/data-source clients (see the shared test doubles below). No real network access, no real backend process involved.
  - **Backend integration** — `@nestjs/testing` + supertest against a real Nest application instance, exercising the real HTTP pipeline (routing, DI, validation pipes); external calls are intercepted with `nock` instead of reaching the real network.
  - **Frontend unit** — Angular's `TestBed` with CDK harnesses, HTTP calls mocked via Angular's own HTTP-testing utilities. No real backend process involved.
  - **Frontend integration** — frontend services/components tested against an actually-running backend process, over real HTTP, verifying the real request/response contract end to end (including SSE parsing) — not mocked at the frontend's own HTTP layer. That backend process is the same kind used for backend integration tests: a real Nest instance with test doubles bound in place of its own external clients, so this never needs a real credential either. This is *not* browser-driven e2e — no simulated clicks or page navigation, just the real network boundary between frontend code and a real backend process.
  - **Frontend browser E2E** — Playwright (`e2e/`, its own top-level directory alongside `frontend/`/`backend/`, not nested under `frontend/` — a different test runner and a heavy, binary-downloading dependency that only needs to run occasionally, not on every `frontend` build) driving a real browser against the actual running dev stack: real page navigation, real clicks, real streaming as a browser actually receives it, unlike Frontend integration's HTTP-only boundary above. Wired as a profile-gated `docker-compose.dev.yml` service (`e2e`) so a bare `... up` never starts it — only `docker compose -f docker-compose.dev.yml run --rm e2e` does (naming a profiled service directly with `run` auto-activates its own profile; no extra `--profile e2e` flag needed). Only ever runs against a fake-mode instance — a `globalSetup` script (`e2e/global-setup.ts`) queries `GET /api/mode` and aborts before any spec runs if `fakeMode` isn't `true`, the one bucket where this rule is actually enforced in code rather than relied on, since it's also the one bucket that runs against a real live app process instead of an isolated module or mocked boundary. One spec file per lab it covers, exercising that lab's own route/happy-path/major-check flow, not exhaustive per-lab coverage.
  - General test-writing mechanics for each stack (how a suite is structured, which harnesses are used) are a coding-convention concern, not covered here — this file is scoped to testing *strategy*: which buckets exist, what each verifies, and how mocking/credentials work.

- **No container that runs tests ever holds a real credential.** `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, and any future data-source key exist only in the developer's own `backend/.env`, used for manually running the app via `docker compose -f docker-compose.dev.yml up` (or the prod equivalent). Anything that builds or runs the test suite — a local `npm test`, a future CI run, or this app being built out during a coding session — uses placeholder values instead, never the real ones. Backend integration/e2e tests boot the real `AppModule`, which enforces config startup validation (see `env-config.md`) — `backend/test/setup-env.ts`, wired via `jest-e2e.json`'s `setupFiles`, supplies a placeholder `ANTHROPIC_API_KEY` before every e2e run so no test file needs its own env setup for this.

- **Startup validation only checks presence, never validity.** The backend's fail-fast startup check for a required credential (missing → a clear startup error) verifies the variable is *set*, not that it's a genuine working key — confirming a real key would need an actual network round-trip, which config validation doesn't do. This is exactly what makes a placeholder string sufficient wherever a real key isn't available: it satisfies "set" without ever being used for a real call, since every external call in a test is mocked before it would reach the network (see below).

- **Every external client sits behind a mockable seam.** The Anthropic SDK client and each data-source client (GitHub, Open-Meteo, arXiv, Wikimedia Commons) are injected via DI, per the existing Nest convention of depending on an interface/token for anything that needs mocking — never `new`'d up inline in a service. This is what makes both backend test levels possible without a real credential or real network access:
  - **Backend unit tests** substitute a fake implementation of the client entirely (a canned response, no HTTP involved at all).
  - **Backend integration tests** let the real client code run (real SDK calls, real Octokit/axios calls), but `nock` intercepts the outbound HTTP request before it leaves the process and returns a canned fixture response — so the integration test still exercises the app's actual request-building and response-handling code, just without a real network round trip. The same real-but-doubled backend process is what frontend integration tests run against, per the bucket above.
  - **A real client's own file exports the external API's raw response shapes for its `nock` fixtures to reuse.** When a `nock` fixtures file needs the shape of what the external API actually returns, that shape is exported from the real client's own file (e.g. `real-github-client.ts`'s `GithubIssueResponse`/`GithubCommitResponse`/etc., `anthropic-client.ts`'s `AnthropicMessage`/`AnthropicStreamEvent`) and imported into the fixtures file, rather than redefined there — one definition of "what the external API actually returns" per client, whether that client is a shared module or a lab-local one.

- **Shared test doubles, not one ad hoc mock per lab.** A fake Anthropic client and fixture responses for each data source live in one shared place (`backend/src/testing/`, see `test-doubles.md`) that every lab's tests reuse, rather than each lab inventing its own — one shared home per concern applies to test doubles the same as it does to any other shared module.

- **Tests exist to catch real regressions, not to pad a coverage number.** Write a test because some specific behavior could plausibly break and would matter if it did — business logic, request/response shaping, error handling, edge cases in a shared module several labs depend on. Skip tests for trivial pass-through code (a plain getter, a DTO's own field assignment, framework wiring with no logic of its own) — a test that can't fail in any way that matters isn't worth writing or maintaining.

- **No feature needs its own "test under both `FAKE_MODE=true` and `FAKE_MODE=false`" matrix.** A feature's business logic is already fully covered by the two backend test buckets above — unit tests inject a fake client directly into the test module, integration tests intercept the real client's outbound call with `nock` — neither of which goes through the `FAKE_MODE` env var; that flag only matters for the live `docker compose up` process. A feature's own `useFactory: config.fakeMode ? fake : real` wiring is trivial pass-through, already covered by "skip tests for... framework wiring with no logic of its own" below, and trusting it means trusting the shared DI-switch helper, which fake mode's own test suite proves once, generically. The real-mode path (a real client actually reaching the real API) stays manually verified either way, same as always — no test ever holds a real credential.

- **`ConfigModule.forRoot()` reads `process.env`/`.env` once, at `AppConfigModule`'s first import in the process — not per `Test.createTestingModule().compile()`.** An e2e spec that imports the real `AppConfigModule` and mutates `process.env.SOME_FLAG` between `it()` blocks, expecting each fresh `compile()` to pick up the new value, silently doesn't work — every compile in that process reuses the snapshot captured at that first import. This is one more reason the previous bullet's guidance holds: prove env-driven DI-switch behavior (e.g. a specific token's `FAKE_MODE=true` vs `false` binding) at the unit level with a stubbed `AppConfigService`, not by toggling `process.env` across e2e test cases. `backend/test/setup-env.ts` pins a deterministic baseline (`ANTHROPIC_API_KEY`, `FAKE_MODE`) for exactly this reason — so e2e specs never depend on, or need to fight, a developer's own `backend/.env`.

- **Backend `npm test` alone doesn't type-check.** `ts-jest` runs with `isolatedModules: true` for fast per-file transpilation, which skips type-checking — a genuine type error can pass the test suite silently. `npm run lint` (type-aware ESLint) is the step that catches it, which is why `CLAUDE.md`'s "Running tests" lists it alongside the test commands, not as an optional extra. The frontend doesn't have this particular gap — `ng test` compiles through the Angular compiler, which type-checks same as `ng build` — but it still has its own `npm run lint` (`@angular-eslint`) for Angular/TypeScript style and template-accessibility rules that no test run covers.

- **CI runs the same buckets, the same way, with the same rule.** `.github/workflows/deploy.yml`'s `test` job (see `demo-deploy.md`) runs the exact lint/unit/integration/browser-E2E commands documented above and in `README.md`, with only placeholder credentials — nothing about this policy changes just because the tests now also run in CI. One CI-specific gotcha worth knowing generally: any command that starts a Compose service participating in a `depends_on: condition: service_healthy` chain (not only the command that's actually testing that service) transitively triggers that whole chain's startup — a placeholder `backend/.env` needs to already exist before the *first* such command in a job, not just before the step that most obviously needs it, or the health-gated service never comes up.
