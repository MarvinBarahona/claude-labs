# Technical — Loading States

Update as the system evolves.

- **A section whose content loading would otherwise make it disappear completely, or expand/shrink the layout drastically, shows skeleton placeholders in that content's place instead of hiding the section.** Toggling a whole section in and out of the DOM around a fetch (or re-fetch) reads as a glitch, not a loading state — especially on a re-fetch that clears previously-shown content, where the section blinks out and back in. Keep the section's container mounted whenever it's either loading or has content, and swap only its inner content between the real result and skeleton placeholders shaped roughly like what's coming (matching line count/aspect for prose, pill shapes for tags, etc. — see `Skeleton`, `frontend/src/app/shared/skeleton/skeleton.ts`) — never an empty gap.

- **A loading state is held for a minimum of 500ms even if the underlying fetch resolves sooner.** Fake mode answers near-instantly, which otherwise makes a skeleton flash by too fast to register as a loading state at all — arguably a worse glitch than not having one. For an Observable-driven fetch, race it against a floor timer: `forkJoin([source$, timer(500)])`, taking the fetch's own result once both resolve. For an imperative flow that isn't a single Observable (e.g. driving a stream reader by hand), record `Date.now()` when loading starts and, immediately before whatever transition would clear the loading state, await the remainder of the 500ms window (`500 - (Date.now() - startedAt)`) if any is left.

- **A minimum-duration loading state is tested by manipulating time, not by waiting on it — and every duration in that test is an expression of one local constant, never a bare number.** The component's own minimum-duration constant isn't exported, so its spec file declares its own same-named local copy (e.g. `MIN_LOADING_MS`) at the top and drives every wait off it: `vi.advanceTimersByTimeAsync(MIN_LOADING_MS)`, `MIN_LOADING_MS - 1` to assert the state still holds just short of the floor, `MIN_LOADING_MS + 100` for a real-time margin. Changing the component's actual duration then only ever means updating that one constant in the spec file to match — never hunting down scattered literals, and never a test name that states the duration in prose (e.g. "...for at least 500ms...") going stale next to it. For the `forkJoin`/`timer` shape, enable `vi.useFakeTimers()` and advance virtual time rather than waiting on a real timer. For the imperative `Date.now()`/`setTimeout` shape layered over other real-Promise-driven async work (a stream reader resolving via direct callback invocation, not timers), fake timers would also have to intercept that other work's own waits, which is fragile — real timers plus a real wait comfortably past the floor (still expressed off the same constant) is the simpler, more robust choice there. Either way, reset with `vi.useRealTimers()` in `afterEach` so one test's fake clock never leaks into the next.
