# Technical — Tech Stack

Update as the system evolves.

- **Frontend:** Angular, latest version.
- **Frontend component library:** Spartan/ng (shadcn-style primitives on Tailwind + Angular CDK) — components are generated into the repo as owned source rather than installed as an opaque package. Chosen over Angular Material for a lighter, more elegant default look, and because AI-assisted edits work directly against the actual component source instead of a black-box dependency. Generating a primitive (`ng g @spartan-ng/cli:ui <name>`) requires `frontend/components.json` to exist first (`componentsPath`, `style`, `importAlias`) — without it the generator prompts interactively and produces nothing under a non-interactive shell. Re-running `ng g @spartan-ng/cli:init` rewrites `package.json`'s `devDependencies` wholesale, which silently drops a manually-added `@spartan-ng/cli` entry — don't reach for it to "fix" a missing dependency.
- **Backend:** NestJS (Node/TypeScript).
- **Backend HTTP adapter:** Express (NestJS's default, `@nestjs/platform-express`) — the broadest ecosystem and what most Nest examples assume; nothing about this app's traffic needs Fastify's extra throughput.
- **Backend env validation:** Zod, via `@nestjs/config`'s `validate` option — one schema library for env validation and for the Claude API SDK's own structured-output/tool-definition helpers (`zodOutputFormat`, `betaZodTool`, both Zod-based) instead of a second one (Joi) used only for config. DTO validation elsewhere still uses class-validator per the existing Nest convention, unaffected by this.
- **Backend external HTTP calls:** `@nestjs/axios` (`HttpService`) for every external data-source call (GitHub, Open-Meteo, arXiv, Wikimedia Commons) — injectable via DI and easy to mock in unit tests, per the convention of putting an external API client behind a DI token rather than calling it directly from a service.
- **Runtime:** both apps are built and run exclusively through Docker; no local Node/Angular/Nest CLI install is assumed, even for a fresh clone. Each project's `node_modules` lives in a Docker-named volume layered over the source bind mount rather than inside the bind mount itself, auto-seeded from the image's own build-time `npm install` the first time a container is created — this keeps `node_modules` off the host filesystem entirely (a plain bind mount would otherwise shadow it with whatever, or nothing, exists on the host) and off the host's I/O path, which measurably matters on a Windows host where a large `node_modules` tree read through a Docker Desktop bind mount is slower and prone to file-rename races. Because the volume persists across restarts, a `package.json` change needs `docker compose down -v` (or removing that volume) alongside a rebuild for new dependencies to actually take effect — a rebuild alone won't refresh an already-populated volume. Both Compose services carry a `healthcheck` against their own smoke-test route/port, with the frontend's `depends_on` gated on the backend's `condition: service_healthy` rather than plain `depends_on` — this waits for the backend to actually finish starting, not just for its container to exist, and gives `docker compose ps` a real `(healthy)` readiness signal. The frontend container runs the Angular CLI dev server (`ng serve`), not a built static bundle — its own dev-server proxy is what connects it to the backend (see `architecture.md`). On a Windows host, the same Docker Desktop bind-mount limitation also breaks `ng serve`'s default file watcher: inotify events from host-side edits under the bind mount don't reliably reach it, so changes silently stop auto-rebuilding until the container is restarted. `frontend/package.json`'s `start` script runs `ng serve --poll 1000` to work around this — polling instead of relying on filesystem change events. A separate production configuration also exists for running (not developing) the app: a compiled Angular production bundle and compiled Nest output, built into a single runtime image where the backend serves the frontend's static build directly — no dev server, no proxy, and dependencies installed straight into the image at build time rather than through a bind-mounted, named-volume `node_modules` (that volume's whole purpose — fast iteration over a live source bind mount — doesn't apply once there's no live-editing use case). `CLAUDE.md` carries the actual `docker compose` commands, dev and prod alike.

Coding conventions for each stack (how the code is written, not what the stack is) aren't tracked here.
