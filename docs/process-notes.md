# Process Notes

Append-only log of suggestions the drafting/build process can't apply itself: coding-convention/process changes, and changes to files it doesn't own (`README.md`, `CLAUDE.md`, other skills). Written during planning or graduation, never applied automatically. Reviewed and cleared manually, on your own schedule.

- **`github-provider`:** when a real client's fixtures file needs the external API's raw response shapes, export those raw shapes from the real client's own file and import them into the fixtures file, rather than redefining them there — keeps one definition of "what the external API actually returns" per client. `real-github-client.ts` exports `GithubIssueResponse`/`GithubCommitResponse`/`GithubReleaseResponse`/`GithubRepoResponse`/`GithubTreeResponse` for exactly this reason, mirroring how `anthropic.fixtures.ts` already reuses `AnthropicMessage`/`AnthropicStreamEvent` from `anthropic-client.ts` instead of duplicating them.
- **`live-tool-use-console`:** `lab-catalog.ts`'s header comment ("populated by the write-lab-doc skill") is only accurate for an entry's wording quality, not for whether the entry must exist. `Home`'s component spreads `LAB_CATALOG[feature.slug]` unguarded for every non-`home` `FEATURE_ROUTES` entry, so a route registered without a matching `LAB_CATALOG` entry throws rather than renders blank — the entry has to exist the moment the route is registered, with `write-lab-doc` only refining its wording later.
- **`live-tool-use-console`:** in a multi-call tool-use loop, reassign the request-params object between iterations rather than mutating it in place. Each `calls` entry stores a snapshot of the exact request sent for that call; mutating the same object after pushing it onto `calls` would retroactively change every earlier entry's `request` to reflect the final mutated state instead of what was actually sent. Worth following for any future multi-call-loop feature.
- **`content-block-builder`:** the exact SDK method name behind the planned `AnthropicClient.uploadFile()` (e.g. `client.beta.files.upload(...)`) wasn't verified against the installed `@anthropic-ai/sdk` — confirm the real call surface when building.
- **`data-code-sandbox`:** the code-execution tool's `container_upload` content-block shape and the Agent Skill registration mechanism (whether a `SKILL.md` package needs a one-time upload/skill ID, or is referenced another way) weren't verified against current Claude API docs during planning — confirm both before wiring the request.
- **`data-code-sandbox`:** "stars over time" was dropped from the plan's dataset (the shared GitHub client has no stargazer-history method) — worth a second look if that visual mattered to you.
- **`document-research-assistant`:** the plan introduces the app's first server-owned, in-memory conversation session (keyed by a generated `sessionId`), a deliberate departure from every earlier lab's stateless-per-request pattern (frontend resends full history). Flagging it since it may become the template later multi-turn features (e.g. a future agent-style lab) copy — worth confirming this is the shape you want before that happens.
- **`workflow-gallery`:** planned as non-streaming, relying on the response's `calls` array to show the full multi-stage pipeline trace after the fact rather than live per-stage progress — worth confirming that reads well once you see the actual UI.
- **`deepwiki-connector`:** the exact DeepWiki MCP server URL and its tool names weren't verified against DeepWiki's own current listing during planning — confirm both when building.
- **`web-repo-research-reporter`:** the web search tool's `max_uses: 5` cap is an arbitrary planning-time default, not a value derived from any real constraint — revisit if a research question routinely wants more (or fewer) searches.
- **`vision-lab`:** the plan's `WikimediaClient` interface (`searchImages(query, count)`, `imageinfo`-derived width/height) wasn't verified against Wikimedia Commons' actual MediaWiki API parameters during planning — confirm the real query shape when building.
- **`extended-thinking-bench`:** the plan deliberately does *not* reuse Workflow Gallery's internal draft-prompt logic — it writes its own independent "draft a response" prompt over the same issue data instead of importing Workflow Gallery's, to avoid a premature shared-module extraction before Workflow Gallery itself is even built. Worth a second look once both features exist, in case the two prompts drift further apart than intended for a fair thinking-on/off comparison.
- **`agent-playground`:** the 10-tool-call iteration cap is an arbitrary planning-time default, not derived from any real constraint — revisit if a real run routinely needs more steps to reach a good answer, or hits the cap too easily. The `search` tool is also scoped to file-*path* substring matching only, deliberately not full-text code search (which would need a separate, rate-limited GitHub API) — flagging in case that's a weaker "search" than you had in mind for the agent to work with.
