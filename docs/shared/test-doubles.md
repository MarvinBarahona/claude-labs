# Test Doubles for External Clients

Shared fakes for every external client the backend talks to тАФ the Anthropic SDK client and each data-source client (GitHub, Open-Meteo, arXiv, Wikimedia Commons) тАФ plus a `nock`-based HTTP-fixture helper for integration tests. No lab writes its own ad hoc mock, and no test anywhere needs a real credential or real network access (see `testing-strategy.md`).

## Interface

All exported from `backend/src/testing/index.ts` (`AnthropicClient` itself re-exported there from its actual build-included location, below, so a consumer can still import it from the same barrel as its fake).

- **Fake Anthropic client** (`backend/src/testing/anthropic/fake-anthropic-client.ts`) тАФ `FakeAnthropicClient` returns canned non-streaming responses, canned streaming event sequences, and canned tool-use loops. Binds to the `AnthropicClient` abstract-class DI token (`createMessage()` / `streamMessage()`), which now lives at `backend/src/shared/anthropic-client/anthropic-client.ts`, not under `testing/` тАФ it has to be build-included so a real adapter's own module can bind it in production. `FakeAnthropicClient` itself stays under `testing/`, per the "Build note" below.
- **`nock`-based fixture helper** (`backend/src/testing/http-fixtures/`) тАФ one fixture set per external host, so integration tests exercise the app's real HTTP-calling code (real SDK/Octokit/axios calls) without any request leaving the test process. Currently covers the Anthropic host only: `mockAnthropicMessagesCreate` / `mockAnthropicMessagesStream` / `mockAnthropicModelsList` and their auth-error variants, plus `useNockFixtures()`, a generic lifecycle helper any host's fixtures can use. `useNockFixtures()` disables real network access for external hosts but always leaves loopback (`127.0.0.1`/`localhost`) connections enabled, so a `supertest`-driven e2e test can still reach the real local app under test in the same run as an external-host mock. No data-source host is covered yet тАФ none is consumed anywhere yet.
- **Data-source fakes** (GitHub, Open-Meteo, arXiv, Wikimedia Commons) тАФ not yet built. Added incrementally, one per data-source client, alongside whichever task/feature first consumes that client тАФ not all up front. Each follows the same pattern as the Anthropic client: `backend/src/testing/<source>/`, a `Fake<Source>Client` bound to an abstract-class DI token (the token itself build-included, per the Anthropic client's own layout above), exported from `backend/src/testing/index.ts`.

## Using it

- **Backend unit tests** тАФ bind the DI token (e.g. `AnthropicClient`) to the fake implementation in the test module instead of a real provider, and get a canned, deterministic response with no HTTP involved.
- **Backend integration tests** тАФ call `useNockFixtures()` for lifecycle setup, then the relevant per-host fixture functions (e.g. `mockAnthropicMessagesCreate`) before exercising the real request path.

## Build note

`backend/src/testing/` is a shared-module folder (per `repo-layout.md`'s "shared functionality" rule) but for test-only code тАФ it's excluded from `backend/tsconfig.build.json`'s build (mirroring how `**/*spec.ts` is already excluded). This keeps `nock` and other test-only *dependencies* out of `dist` reliably, since nothing build-included imports `testing/http-fixtures/` (nock's only consumer). It does not guarantee no file under `testing/` is ever emitted, though: TypeScript's `exclude` only keeps a folder out of the initial build-root file set тАФ it does not stop a file from being compiled and emitted if a build-included file imports it. `AnthropicClientModule` (build-included) statically imports `FakeAnthropicClient` to bind it via `fakeSwitchProvider`, so `dist/testing/anthropic/fake-anthropic-client.js` is in fact present in a production build. This is an accepted tradeoff, not a bug: `FakeAnthropicClient` itself has no test-only dependencies (no `nock`, no `jest`), so nothing heavier than one small pure-TS class leaks in. A future data-source fake wired the same way should keep this in mind тАФ if a fake ever needs a genuinely test-only dependency, that dependency would leak into `dist` too.
