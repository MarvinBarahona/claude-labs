# Test Doubles for External Clients

Shared fakes for every external client the backend talks to — the Anthropic SDK client and each data-source client (GitHub, Open-Meteo, arXiv, Wikimedia Commons) — plus a `nock`-based HTTP-fixture helper for integration tests. No lab writes its own ad hoc mock, and no test anywhere needs a real credential or real network access (see `testing-strategy.md`).

## Interface

All exported from `backend/src/testing/index.ts` (`AnthropicClient` itself re-exported there from its actual build-included location, below, so a consumer can still import it from the same barrel as its fake).

- **Fake Anthropic client** (`backend/src/testing/anthropic/fake-anthropic-client.ts`) — `FakeAnthropicClient` returns canned non-streaming responses, canned streaming event sequences, canned tool-use loops, canned file-upload/download results, and canned skill-registration results. Binds to the `AnthropicClient` abstract-class DI token (`createMessage()` / `streamMessage()` / `uploadFile()` / `downloadFile()` / `registerSkill()`), which now lives at `backend/src/shared/anthropic-client/anthropic-client.ts`, not under `testing/` — it has to be build-included so a real adapter's own module can bind it in production. `FakeAnthropicClient` itself stays under `testing/`, per the "Build note" below.
  - With nothing queued, a call throws by default — a test that forgot to queue a response fails loudly rather than silently asserting on the wrong thing. The public `allowUnqueuedFallback` flag (default `false`) flips this to a generic canned response instead (`uploadFile()`'s canned fallback is `{ id: 'file_fake_unqueued' }`, `downloadFile()`'s is a canned buffer/mediaType/filename, `registerSkill()`'s is `{ id: 'skill_fake_unqueued' }`); `AnthropicClientModule` (see `anthropic-client.md`) enables it only on the instance it binds for a live, fake-mode-running app, so an unscripted click in the running app never dead-ends in a 500. Leave it at its default in tests.
  - The unqueued-call fallback recognizes three request shapes and tailors its canned response accordingly, identically on both `createMessage()`'s and `streamMessage()`'s fallback path (the code-execution shape below is `createMessage()`-only, since no streaming lab offers that tool yet): a schema-less tool offered in `tools` (`'type'` starting with `text_editor_`, no `input_schema` — a custom JSON-schema tool's own `description` field is what the plain tool-call fallback otherwise relies on, so a schema-less tool needs this separate check) gets a synthesized `create` call against a fixed `/notes.md` path; a `document` content block carrying `citations: { enabled: true }` anywhere in `messages` gets one fabricated `page_location` citation attached to the fallback's text answer; a `tools` entry whose `type` starts with `code_execution_` (a server-executed tool, so there's no round trip to gate on the way the client-tool shapes above do) gets a fabricated `server_tool_use`(`bash_code_execution`)/`bash_code_execution_tool_result` pair with a fabricated output `file_id`, so a code-execution lab's unscripted click has real executed-code and output-file content to render instead of plain text. All three checks run independent of streaming vs. non-streaming where both paths apply, so an unscripted click against a citations-enabled or text-editor-tool-offering lab exercises the same fallback behavior whichever mode it asked for.
  - `queueMessage()` / `queueStream()` / `queueFileUpload()` / `queueFileDownload()` / `queueSkillRegistration()` each push onto their own FIFO queue, consumed in call order by `createMessage()` / `streamMessage()` / `uploadFile()` / `downloadFile()` / `registerSkill()` respectively.
- **`nock`-based fixture helper** (`backend/src/testing/http-fixtures/`) — one fixture set per external host, so integration tests exercise the app's real HTTP-calling code (real SDK/Octokit/axios calls) without any request leaving the test process. Covers the Anthropic host (`mockAnthropicMessagesCreate` / `mockAnthropicMessagesStream` / `mockAnthropicModelsList` / `mockAnthropicFilesUpload` / `mockAnthropicFilesRetrieveMetadata` / `mockAnthropicFilesDownload` / `mockAnthropicSkillsCreate` and their auth-error variants) and the GitHub host ([`github-provider.md`](github-provider.md)'s `mockGithubIssues` / `mockGithubCommits` / `mockGithubReleases` / `mockGithubRepo` / `mockGithubTree` and their `mockGithubRateLimitError` / `mockGithubNotFoundError` error variants), plus `useNockFixtures()`, a generic lifecycle helper any host's fixtures can use. `useNockFixtures()` disables real network access for external hosts but always leaves loopback (`127.0.0.1`/`localhost`) connections enabled, so a `supertest`-driven e2e test can still reach the real local app under test in the same run as an external-host mock. Open-Meteo, arXiv, and Wikimedia Commons aren't covered yet — none is consumed anywhere yet.
- **Data-source fakes** (GitHub, Open-Meteo, arXiv, Wikimedia Commons) — added incrementally, one per data-source client, alongside whichever task/feature first consumes that client — not all up front. [`github-provider.md`](github-provider.md)'s `FakeGithubClient` is the first one built, and follows the same pattern as the Anthropic client: `backend/src/testing/<source>/`, a `Fake<Source>Client` bound to an abstract-class DI token (the token itself build-included, per the Anthropic client's own layout above), exported from `backend/src/testing/index.ts`. Unlike `FakeAnthropicClient`, `FakeGithubClient` has no unqueued-call fallback flag — GitHub data is naturally static, so it returns built-in canned fixtures by default instead of throwing when nothing's queued; a future data-source fake with similarly static data can follow this variant instead of the throw-when-unqueued one.

## Using it

- **Backend unit tests** — bind the DI token (e.g. `AnthropicClient`) to the fake implementation in the test module instead of a real provider, and get a canned, deterministic response with no HTTP involved.
- **Backend integration tests** — call `useNockFixtures()` for lifecycle setup, then the relevant per-host fixture functions (e.g. `mockAnthropicMessagesCreate`) before exercising the real request path.

## Build note

`backend/src/testing/` is a shared-module folder (per `repo-layout.md`'s "shared functionality" rule) but for test-only code — it's excluded from `backend/tsconfig.build.json`'s build (mirroring how `**/*spec.ts` is already excluded). This keeps `nock` and other test-only *dependencies* out of `dist` reliably, since nothing build-included imports `testing/http-fixtures/` (nock's only consumer). It does not guarantee no file under `testing/` is ever emitted, though: TypeScript's `exclude` only keeps a folder out of the initial build-root file set — it does not stop a file from being compiled and emitted if a build-included file imports it. `AnthropicClientModule` (build-included) statically imports `FakeAnthropicClient` to bind it via `fakeSwitchProvider`, so `dist/testing/anthropic/fake-anthropic-client.js` is in fact present in a production build. This is an accepted tradeoff, not a bug: `FakeAnthropicClient` itself has no test-only dependencies (no `nock`, no `jest`), so nothing heavier than one small pure-TS class leaks in. A future data-source fake wired the same way should keep this in mind — if a fake ever needs a genuinely test-only dependency, that dependency would leak into `dist` too.
