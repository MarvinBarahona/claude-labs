# App Shell

The shared Angular routing, top-level layout, and navigation chrome every lab plugs into: one route per feature, a persistent header/content frame, and a left-sidebar nav — plus the page-composition convention every lab's own route component follows.

## Interface

- **Feature registry** (`frontend/src/app/core/feature-registry.ts`): `FEATURE_ROUTES`, an ordered array of `{ slug, label, loadComponent }`. Array order *is* the nav render order — there is no `navPosition` field or ordering function anywhere in the app. A feature's own plan file records a `**Nav position:**` line (`first` / `last` / `before <slug>` / `after <slug>`) as planning-time guidance only; whoever registers the feature inserts its entry at the corresponding index in `FEATURE_ROUTES` by hand, once.
- **Routing** (`frontend/src/app/core/build-feature-routes.ts`): `buildFeatureRoutes(features)` builds one lazy-loaded child route (`loadComponent`, never an eager `component`) per feature, keyed by its slug, wrapped under a single root route whose `component` is `Layout` and whose `data` carries `{ features }` (consumed via `provideRouter(routes, withComponentInputBinding())` in `app.config.ts`, so `Layout`'s `features` input always matches whatever list built the routes). The root path (`/`) redirects to the first feature in `FEATURE_ROUTES`; when the registry is empty, that redirect child route is omitted entirely rather than emitting a self-redirecting `redirectTo: ''`. The `FeatureRoute` type lives in `frontend/src/app/core/feature-route.ts`, kept separate from `build-feature-routes.ts` (which needs `Layout`) specifically to avoid a circular import — `Layout` imports `Nav`, which needs the `FeatureRoute` type; keep this split if this area is touched again.
- **`Layout`** (`frontend/src/app/shared/layout/layout.ts`, selector `app-layout`): persistent header plus a `<router-outlet>` content area. Takes a `features` input (`readonly FeatureRoute[]`, default `[]`) and renders `<app-nav [features]="features()" ...>` alongside the outlet. Owns a `navOpen` signal for the mobile nav overlay (below).
- **`Nav`** (`frontend/src/app/shared/nav/nav.ts`, selector `app-nav`): renders one link per feature, in the given array order, marking the active route distinctly via `routerLinkActive`. Inputs: `features` (`input.required<readonly FeatureRoute[]>()`), `open` (`input(false)` — whether it's shown below the `lg` breakpoint). Output: `linkClick` (emitted when any link is clicked, so a consumer can close a mobile overlay on navigation).
- **Mobile nav overlay:** below the `lg` breakpoint, `Nav` is `hidden` unless its `open` input is `true`, in which case it renders as a `fixed` overlay (`inset-y-0 left-0 z-40`) instead of the static sidebar column. `Layout` owns the `navOpen` signal, a `lg:hidden` header toggle button (inline SVG hamburger, no icon library dependency), and a `lg:hidden` backdrop (`fixed inset-0 bg-black/50`) that closes the nav on tap; `Nav`'s `linkClick` output also closes it, so selecting a lab collapses the overlay automatically. Above `lg`, the sidebar is always visible and none of this applies.
- **Docs panel wiring:** a feature's route component binds `[slug]="'<feature-slug>'"` (or resolves it from route data) on `<app-docs-panel>` — no further plumbing needed; see `docs-panel.md`.
- **Lab page composition convention:** a feature's route component stacks its panels in a single column, top-to-bottom, in this order: `DocsPanel` first (the concept/data-source framing), then the lab's own interactive demo, then `InspectorPanel` last (what actually happened, after acting). This favors a narrative reading order (explain, try, inspect) over a permanent side-by-side reference layout — this app is a hands-on companion for trying the Claude API, not a replacement for the official API docs, so the docs panel doesn't need to stay pinned as a live reference while the demo runs. Adapting a lab's own demo/docs/inspector content to small screens beyond this stacking (e.g. any lab-specific responsive behavior) is each feature's own responsibility, not this shell's.

## Using it

To add a graduated feature to the app: insert `{ slug, label, loadComponent: () => import('<feature-path>').then((m) => m.<FeatureComponent>) }` into `FEATURE_ROUTES`, at the array index its own plan file's `**Nav position:**` line calls for. Nothing else needs to change — routing, nav rendering, and the root redirect all derive from that one array. The feature's own route component then follows the docs → demo → inspector stacking convention above.

## Testing

- `frontend/src/app/shared/nav/nav.spec.ts` — link order follows the given `features` array; hidden by default and shown when `open` is `true`; emits `linkClick` on selection.
- `frontend/src/app/shared/layout/layout.spec.ts` — via `RouterTestingHarness` against mock feature routes: persistent header/chrome renders around a feature route with the content outlet swapping per route; root redirects to the first feature; active link is marked distinctly; a feature route is lazy-loaded (`loadComponent`, not `component`); the header toggle button opens the mobile nav overlay and it closes again after selecting a link.
- `frontend/src/app/core/build-feature-routes.spec.ts` — each feature is keyed by slug behind `loadComponent`; the root redirect targets the first feature in array order; the redirect is omitted when the registry is empty.
