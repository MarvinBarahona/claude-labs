# Chat Transcript

A shared frontend building block for the chat-style, multi-turn Q&A "Transcript" pattern used by both Messages Console and Document Research Assistant — a scrollable, bordered bubble list with markdown rendering, a pending-turn skeleton, an empty-state placeholder, and an input dock, so a lab's own transcript code is limited to supplying turns/streamed text and reacting to a `send` event.

## Interface

`frontend/src/app/shared/chat-transcript/` (`ChatTranscript`, selector `app-chat-transcript`):

- **Turn model** — exported `ChatTranscriptTurn = { question: string; answerMarkdown: string | null }` (`answerMarkdown: null` means this turn's answer hasn't landed yet). At most one turn has `answerMarkdown === null` at a time — the most recently sent one.
- **Inputs:** `turns: readonly ChatTranscriptTurn[]`; `pendingAnswerMarkdown: string | null` (the in-progress streamed text for the one pending turn, if any); `disabled: boolean` (Send button disabled, and the input switches to read-only, while a turn is in flight — kept off the `<input>`'s own `disabled` attribute deliberately, since a genuinely `disabled` HTML input is force-blurred by the browser the instant it becomes disabled, which would kick focus out of the input on every send); `placeholder: string`; `ariaLabel: string`; `sendLabel: string` (default `'Send'`; Document Research Assistant passes `'Ask'`); `answerBodyTemplate: TemplateRef<{ $implicit: ChatTranscriptTurn; index: number }> | null` (default `null`) — a per-turn custom body slot for a consumer needing more than plain markdown in the assistant bubble (Document Research Assistant's citation markers). It only applies once a turn's `answerMarkdown` is non-null — the pending/skeleton states always render generically, regardless of whether a custom template is supplied. The context's `index` lets a consumer look up its own richer per-turn data (e.g. citations) from its own array by position, since the shared turn model itself only carries `question`/`answerMarkdown`.
- **Output:** `send` emits the trimmed draft text on Enter or Send click; the component owns its own draft-text signal, clearing it on emit.
- **Rendering:** a bordered, height-capped scroll box (`max-h-[60vh] overflow-y-auto rounded-md border border-border bg-background p-3`) containing an `<ol data-testid="transcript-list">` (one `<li>` per turn, right-aligned user bubble + left-aligned assistant bubble). An empty-state placeholder (`data-testid="transcript-empty"`, `"No messages yet — say something to get started."`) shows above the list when there are no turns yet. The scroll box auto-scrolls to the bottom whenever `turns()`/`pendingAnswerMarkdown()` change (a `viewChild`-driven `effect()`), keeping the newest turn in view — it does not preserve a user's own scroll-up position while reading older history, if they've scrolled away when a new turn lands. The assistant bubble shows, in priority order: once `answerMarkdown` isn't null, the custom body slot's content if supplied, else `renderMarkdown(answerMarkdown)` via `[innerHTML]`; else, for the pending turn, `renderMarkdown(pendingAnswerMarkdown)` when non-empty; else a skeleton placeholder (`data-testid="answer-skeleton"`, one wide `app-skeleton` bar). The input dock (text input + Send button, Enter-to-send, empty-trim-to-disable Send) sits below the scroll box.

`frontend/src/app/shared/markdown/render-markdown.ts` — exports `renderMarkdown(text: string): string`, wrapping `marked.parse(text, { async: false })`. The one shared home for markdown rendering — used by `ChatTranscript`'s own default bubble rendering, Document Research Assistant's per-paragraph citation template, and its Notes panel.

## Using it

Import `ChatTranscript` (and `ChatTranscriptTurn` for typing) into a lab's own standalone component. Maintain the lab's own turn/pending-text state (typically a `signal<readonly ChatTranscriptTurn[]>` and a `signal<string>` for streamed text), push a new `{ question, answerMarkdown: null }` turn immediately when sending — before the request resolves, so the pending/skeleton state has a turn to attach to — and update that turn's `answerMarkdown` once the response lands. Handle the `(send)` event to fire the actual request; pair it with a minimum-duration floor timer (per `loading-states.md`) so a near-instant fake-mode response doesn't make the pending skeleton flash by unreadably.

A consumer needing more than plain markdown per turn (citations, etc.) supplies an `answerBodyTemplate` via a local `<ng-template #foo let-turn let-i="index">`, reading its own richer per-turn data by `index` from its own array — not from the `ChatTranscriptTurn` itself, which doesn't carry it.

## Testing

- `frontend/src/app/shared/chat-transcript/chat-transcript.spec.ts` — one `<li>` per turn with correct bubble alignment; markdown rendering of a completed turn's `answerMarkdown`; markdown rendering of `pendingAnswerMarkdown` for the pending turn; the skeleton placeholder when neither is available; the empty-state placeholder appearing/disappearing with `turns().length`; a consumer-supplied `answerBodyTemplate` overriding the default rendering; the input staying focusable (read-only, not `disabled`) while `disabled` is `true`; Send's disabled state (empty/whitespace draft, or `disabled`); Enter/Send both emitting the trimmed draft and clearing it.
- `frontend/src/app/shared/markdown/render-markdown.spec.ts` — markdown-to-HTML conversion (e.g. `**bold**` → `<strong>`); returns a plain string, not a Promise.
- `frontend/src/app/messages-console/messages-console.spec.ts` and `frontend/src/app/document-research-assistant/document-research-assistant.spec.ts` — each lab's own send flow and transcript rendering against this component, including (for Messages Console) the pending-turn skeleton and minimum-duration floor timer it gained by adopting it.
- `e2e/tests/messages-console.spec.ts` / `e2e/tests/document-research-assistant.spec.ts` (Playwright) — each lab's own nav/docs/turn-completion smoke test against the shared component's actual DOM shape (one `<li>` per turn, not per message).
- The scroll/auto-scroll/height-cap behavior itself has no automated coverage — `jsdom` (this project's frontend-unit test environment) doesn't perform real layout, so `scrollHeight` is always `0` and any such assertion would be vacuous; verified manually instead.
