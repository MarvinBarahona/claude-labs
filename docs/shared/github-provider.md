# GitHub Data Provider

A backend module wrapping the GitHub REST API (`api.github.com`) — issues, commits, releases, file tree — against a single configured target repo. Every feature that needs repo data depends on this instead of calling GitHub directly, so there's exactly one place handling GitHub's rate limits and response shapes. This is the app's embodiment of [`guiding-principles.md`](../technical/guiding-principles.md)'s "Minimize integrations" principle: one shared, key-free GitHub integration reused across features, rather than each rolling its own.

## Interface

`backend/src/shared/github-provider/`:

- **`github-provider.types.ts`** — the typed shapes every method returns: `GithubIssue` (`number`, `title`, `state: 'open' | 'closed'`, `body: string | null`, `user`, `createdAt`, `url`), `GithubCommit` (`sha`, `message`, `author`, `date`, `url`), `GithubRelease` (`tagName`, `name: string | null`, `body: string | null`, `publishedAt`, `url`), `GithubFileTreeEntry` (`path`, `type: 'blob' | 'tree'`, `sha`).
- **`GithubClient`** (`github-client.ts`) — the abstract-class DI token: `getIssues(params?: { state?, perPage? })`, `getCommits(params?: { perPage? })`, `getReleases(params?: { perPage? })`, `getFileTree()`, `getFileContent(path: string)`, all `Promise`-returning. Every method operates on the single configured target repo ([`env-config.md`](env-config.md)'s `AppConfigService.githubTargetRepo`) — no method takes a repo override.
- **`RealGithubClient`** (`real-github-client.ts`) — calls `https://api.github.com` via `axios`, reading `AppConfigService.githubTargetRepo`/`githubToken` via constructor injection. Sends `Authorization: Bearer <githubToken>` when set, no auth header otherwise. `getFileTree()` first reads `GET /repos/{owner}/{repo}` for `default_branch`, then `GET /repos/{owner}/{repo}/git/trees/{default_branch}?recursive=1`, mapping its `tree` array to `GithubFileTreeEntry[]`. `getIssues`/`getCommits`/`getReleases` call their corresponding list endpoints (`/issues`, `/commits`, `/releases`) directly. `getFileContent(path)` calls `GET /repos/{owner}/{repo}/contents/{path}` (each path segment individually URI-encoded); GitHub always returns the file's `content` base64-encoded regardless of whether it's actually text or binary, so this method attempts a strict UTF-8 decode (`TextDecoder('utf-8', { fatal: true })`) and returns `{ content: <decoded text>, encoding: 'utf-8' }` when that succeeds, or `{ content: <the original base64 string>, encoding: 'base64' }` when it doesn't (a genuinely binary file). Any thrown axios error (4xx/5xx, network failure) is caught and rethrown as [`api-error-handling.md`](api-error-handling.md)'s `ExternalApiError('github', <original message>)` — including a not-found path, same as every other method; a caller that needs "not found" treated as a recoverable outcome rather than a transport failure handles that distinction itself; `getFileContent` has no `null`-return convention the way some other data sources' single-item lookups do. The raw GitHub REST response shapes (`GithubIssueResponse`, `GithubCommitResponse`, `GithubReleaseResponse`, `GithubRepoResponse`, `GithubTreeResponse`, `GithubContentResponse`) are exported from this file too, so the `nock` fixtures below can build accurate typed payloads instead of duplicating the shapes.
- **`GithubProviderModule`** (`github-provider.module.ts`) — binds `GithubClient` via [`fake-mode.md`](fake-mode.md)'s `fakeSwitchProvider<GithubClient>(GithubClient, { real: RealGithubClient, fake: FakeGithubClient })`, generic pinned explicitly since `RealGithubClient` has `private` fields `FakeGithubClient` doesn't share. `exports: [GithubClient]`. A consuming feature module imports this and injects `GithubClient`, never `RealGithubClient`/`FakeGithubClient` directly.
- **`FakeGithubClient`** (`backend/src/testing/github/fake-github-client.ts`) — unlike [`test-doubles.md`](test-doubles.md)'s `FakeAnthropicClient` throw-when-unqueued design, GitHub data is naturally static: `FakeGithubClient` returns built-in canned fixture data for all five methods by default, so a live fake-mode app never dead-ends on an unscripted call — no `allowUnqueuedFallback` flag needed. Exposes `setIssues()`/`setCommits()`/`setReleases()`/`setFileTree()`/`setFileContent()` letting a unit test override any one before exercising a consumer; `getFileContent()` ignores its `path` argument and always returns whichever content is currently set (or the built-in default), the same "one canned value regardless of input" shape as the other methods. Exported (alongside a re-export of `GithubClient` from its real, build-included location) from `backend/src/testing/index.ts`.
- **`nock` fixtures** (`backend/src/testing/http-fixtures/github.fixtures.ts`) — `mockGithubIssues`/`mockGithubCommits`/`mockGithubReleases`/`mockGithubRepo`/`mockGithubTree`/`mockGithubContent`, each taking the target repo path plus a raw-shaped payload, and `mockGithubRateLimitError`/`mockGithubNotFoundError`/`mockGithubContentNotFoundError` for the 403/404 error cases. Exported from `backend/src/testing/http-fixtures/index.ts` alongside the Anthropic host's fixtures.

## Using it

- Import `GithubProviderModule` into a feature module and inject `GithubClient` — `FAKE_MODE` governs which implementation is actually live in a running process, nothing feature-specific to configure.
- A failed real call always surfaces as `ExternalApiError('github', ...)`, which `api-error-handling.md`'s global filter shapes into the documented `502` response automatically.
- A unit test binds `FakeGithubClient` directly (optionally calling its setters first); an integration test calls `useNockFixtures()` then the relevant `github.fixtures.ts` functions before exercising `RealGithubClient`'s real request path.

## Testing

- `backend/src/shared/github-provider/github-provider.module.spec.ts` — proves the module binds `FakeGithubClient` when `fakeMode` is `true` and `RealGithubClient` when `false`, using a stub config module.
- `backend/src/testing/github/fake-github-client.spec.ts` — each of the five methods returns its built-in canned fixture by default, and the matching setter overrides it.
- `backend/src/shared/github-provider/real-github-client.spec.ts` — `nock`-intercepted: shapes issues/commits/releases/file-tree responses into their typed arrays, re-points every method's request path when the target repo is overridden, sends no `Authorization` header when `githubToken` is unset and `Authorization: Bearer <token>` when it is, decodes a base64 text file's content to UTF-8 and falls back to reporting `encoding: 'base64'` for genuinely binary content, URI-encodes nested path segments, and rethrows a 403/404 GitHub error response (including a not-found content lookup) as `ExternalApiError('github', ...)`.
